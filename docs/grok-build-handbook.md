## 4. PORTAL v1 design system

**Source of truth:** `src/app/portal-tokens.css` (CSS vars) +
`src/lib/design-system.ts` (TS exports) + `.claude/rules/portal-design-system.md`.

### Core rule: tokens over hex, always

```tsx
// ✅ CORRECT
color: 'var(--portal-fg-1)'
background: 'var(--portal-ink-2)'
border: '1px solid var(--portal-line-2)'
fontSize: 'var(--portal-fs-sm)'

// ❌ WRONG — ratcheted, will fail gates
color: '#E8EAED'
background: 'rgba(0,0,0,0.4)'
fontSize: 13
```

Exceptions: `opengraph-image.tsx` and `twitter-image.tsx` need inline
hex because edge runtime can't resolve CSS vars during rasterization.
These paths are grep-excluded in `scripts/gsd-verify.sh`.

### Token families (the big five)

```
--portal-ink-0..5       Canvas + raised surfaces (page bg → card → hover)
--portal-fg-1..5        Text (brightest → dimmest silver)
--portal-line-1..3      Hairlines (6% → 16% white alpha)
--portal-r-1..5 + r-pill  Radii (4/8/12/16/24px + pill)
--portal-fs-tiny..3xl   Font sizes (11 → 56px)
--portal-font-display   Instrument Serif — the wordmark typeface
--portal-font-mono      Geist Mono — every number, ID, timestamp
--portal-font-sans      Geist — body copy
```

### Semantic status tokens

```
--portal-status-{green,amber,red}-{bg,fg,ring}
--portal-status-green-bg  rgba(34,197,94,0.14)
--portal-status-green-fg  #4ADE80   (visible on canvas)
--portal-status-green-ring rgba(34,197,94,0.30)  (borders)
```

Rule: one semantic color per screen, paired with silver chrome.
Never emerald + red + amber simultaneously in the same view.

### Glass chemistry (three tiers)

```
<GlassCard tier="hero">       rgba(0,0,0,0.4)   — KPI tiles, primary nav
<GlassCard tier="secondary">  rgba(0,0,0,0.25)  — section cards
<GlassCard tier="tertiary">   rgba(0,0,0,0.12)  — chips, drawers
```

All three: `backdrop-filter: blur(20px) saturate(1.2)`.
Never write inline glass chemistry. Always use the primitive.

**Never reimplement any of these inline.** If a primitive is missing a
prop you need, extend the primitive — don't fork it. If you think you
need a new primitive, check `/admin/design` gallery first; often the
shape already exists.

### Extending a primitive

Every primitive follows the pattern:

```tsx
'use client'   // only if it has interactivity
interface Props { … }
export function AguilaThing({ … }: Props) { … }
// Barrel export in src/components/aguila/index.ts
```

When you add props, update the type + re-export:

```ts
// src/components/aguila/index.ts
export { AguilaStagePills } from './AguilaStagePills'
export type { AguilaStageOption } from './AguilaStagePills'
```

## 6. Invariants that never bend

These are load-bearing. Violating them breaks the business, not just
the build. `.claude/rules/core-invariants.md` is the full list —
36 invariants. The top ones:

| # | Rule | Why |
|---|---|---|
| I1 | No hardcoded opaque backgrounds | Every card uses `GlassCard` |
| I7 | Pedimento numbers keep spaces | `DD AD PPPP SSSSSSS` — strip-spaces breaks every SAT lookup |
| I8 | Fracciones keep dots | `3901.20.01` — strip-dots breaks tariff lookups |
| I9 | IVA base = valor_aduana + DTA + IGI | NEVER `invoice × 0.16` flat |
| I11 | Deadlines in `America/Chicago` | Off-by-1h = real regulatory exposure |
| I12 | RLS on every new table | Tested in migration. No exceptions. |
| I13 | No `'9254'` literal in production queries | Use `session.companyId` |
| I17 | Rates from `system_config`, never hardcoded | `getDTARates()`/`getExchangeRate()` — if `valid_to < today` the pipeline refuses to calculate |
| I18 | No silent failures | Every cron fires Telegram on failure |
| I24 | Client surfaces show certainty, not anxiety | No MVE countdowns on `/inicio` |
| I26 | No inline glass chrome outside `src/components/aguila/` | |
| I32 | `audit_log` is the canonical activity-feed source | Not `workflow_events` |
| I34 | Cockpit SSR queries are soft-wrapped | `softCount`/`softData`/`softFirst` from `src/lib/cockpit/safe-query.ts` |

The tenant-isolation contract (`.claude/rules/tenant-isolation.md`) is
HARD. Read §9 before touching anything tenant-scoped.

## 7. How to add a new admin page

Template (admin-only, reads a lead, shows detail, supports filters):

```tsx
// src/app/admin/my-feature/page.tsx
import { cookies } from 'next/headers'
import { redirect } from 'next/navigation'
import { verifySession } from '@/lib/session'
import { createServerClient } from '@/lib/supabase-server'
import { PageShell, GlassCard } from '@/components/aguila'

export const dynamic = 'force-dynamic'
export const revalidate = 30  // or 0 if live data

export default async function MyFeaturePage() {
  const cookieStore = await cookies()
  const session = await verifySession(cookieStore.get('portal_session')?.value ?? '')
  if (!session) redirect('/login')
  if (!['admin', 'broker'].includes(session.role)) redirect('/')

  const supabase = createServerClient()
  const { data } = await supabase.from('…').select('…').limit(200)

  return (
    <PageShell title="My Feature" subtitle="What it does" maxWidth={1400}>
      <GlassCard tier="hero" padding={20}>
        {/* content */}
      </GlassCard>
    </PageShell>
  )
}
```

Add to the nav (if the feature deserves a nav slot) in
`src/components/nav/nav-config.ts` — but only after Tito + Renato IV
sign-off on the slot.

Every admin route also needs `loading.tsx` + `error.tsx` (invariant #36
— cockpit routes specifically, but apply to admin routes too when
they're data-dependent).

## 9. How to add a new API endpoint

Canonical response shape (every endpoint):

```ts
{ data: T | null, error: { code: string, message: string } | null }
```

Error codes: `NOT_FOUND · VALIDATION_ERROR · UNAUTHORIZED · FORBIDDEN ·
RATE_LIMITED · CONFLICT · INTERNAL_ERROR`.

Template (admin-only mutating endpoint):

```ts
// src/app/api/my-resource/[id]/route.ts
import { NextResponse } from 'next/server'
import { cookies } from 'next/headers'
import { verifySession } from '@/lib/session'
import { createServerClient } from '@/lib/supabase-server'

export const runtime = 'nodejs'
export const dynamic = 'force-dynamic'

async function requireAdminSession() {
  const cookieStore = await cookies()
  const session = await verifySession(cookieStore.get('portal_session')?.value ?? '')
  if (!session || !['admin', 'broker'].includes(session.role)) return null
  return session
}

export async function POST(req: Request, ctx: { params: Promise<{ id: string }> }) {
  const session = await requireAdminSession()
  if (!session) {
    return NextResponse.json(
      { data: null, error: { code: 'UNAUTHORIZED', message: 'admin/broker only' } },
      { status: 401 },
    )
  }
  const { id } = await ctx.params
  // 1. Parse + validate body
  // 2. Call lib/ logic (never inline business logic in handlers)
  // 3. Return canonical shape
}
```

**CSRF:** All mutating endpoints require the `X-CSRF-Token` header
(cookie double-submit). Exempt lists live in `src/lib/csrf.ts`. Only
add to exempt list for public (prospect-facing) endpoints — and when
you do, mitigate with honeypot + sanitization + RLS deny-all.

**Business logic goes in `lib/`, never in the route handler.** The
handler is an HTTP adapter: auth, parse, call lib, return.

## 10. How to add a new Supabase migration

```bash
# 1. Create the SQL file
cd ~/evco-portal
NEW_NAME="$(date +%Y%m%d%H%M%S)_description.sql"
touch "supabase/migrations/$NEW_NAME"
# edit — include ENABLE RLS + policy in the same file

# 2. Apply to prod
npx supabase db push

# 3. Regenerate types (stderr-safe form)
npx supabase gen types typescript --linked 2>/dev/null > types/supabase.ts

# 4. Typecheck + test
npx tsc --noEmit
npx vitest run
```

Every migration:
- `IF NOT EXISTS` / `IF EXISTS` for idempotency (re-runnable on dev)
- `ENABLE ROW LEVEL SECURITY` on every new tenant-scoped table
- `FOR ALL USING (false)` policy — portal uses HMAC session + service
  role to bypass. JWT-claim policies always evaluate false because
  we don't use Supabase auth.
- `COMMENT ON TABLE` with one-line purpose
- Write a migration-contract test in
  `src/lib/<domain>/__tests__/<name>-migration.test.ts` — greps the
  SQL file for expected shape. See
  `src/lib/leads/__tests__/migration.test.ts` for the pattern.

## 11. Tenant isolation (READ THIS)

Full contract: `.claude/rules/tenant-isolation.md`. The non-negotiable:

**Every write to a tenant-scoped table MUST include `company_id`.**

Tenant-scoped tables include: `globalpc_productos`, `globalpc_partidas`,
`leads`, `lead_activities`, `traficos`, `entradas`,
`expediente_documentos`, `pedimento_ocas`, `mensajeria_*`,
`operator_actions`, `agent_decisions`, `notifications`.

### Reading from tenant-scoped tables

```ts
const { data } = await supabase
  .from('globalpc_productos')
  .select('*')
  .eq('company_id', session.companyId)   // ALWAYS filter
```

For the catalog (parts that might be cross-tenant), additionally:

```ts
const verifiedCves = activeCvesArray(
  await getActiveCveProductos(supabase, session.companyId)
)
productoQuery = productoQuery
  .eq('company_id', session.companyId)
  .in('cve_producto', verifiedCves)     // belt + suspenders
```

### Writing

Sync scripts MUST stamp `company_id` on every insert. No `'unknown'`
fallback. No `'evco'` default. If the MySQL row's `cve_cliente` isn't
in the allowlist, SKIP the row + fire Telegram.

### Admin sessions

`session.companyId === 'admin'` or `'internal'` for broker/admin
roles. These pseudo-values don't match any real `company_id`, so
admin queries intentionally bypass the tenant filter (owner cockpit,
`/admin/eagle`, aggregates across all clients). Client role surfaces
must always filter by `session.companyId`.

## 12. The lead → client conversion flow (M4)

Prospect journey:
```
/pitch → fills form → POST /api/leads → /admin/leads ("new" stage)
       → sales works the lead (activities logged on every touch)
       → stage moves to "won"
       → broker clicks "Convertir a cliente" in /admin/leads/[id]
       → POST /api/leads/[id]/convert
         · Inserts `companies` row (tenant created)
         · Stamps lead.client_code_assigned + converted_at
         · Emits 'system' activity on lead timeline
       → Lead row is now the provenance link back to the prospect
```

Conversion guardrails:
- Idempotent (won't double-convert)
- Unique (can't overwrite existing tenant)
- Rolls back `companies` insert if lead update fails
- Slug validated: `[a-z0-9-]`, 3-40 chars, no consec dashes

## 13. Testing + gates

Every commit runs `.husky/pre-commit` which checks:
- `npx tsc --noEmit`
- No `CRUD` strings in src/
- No hardcoded client IDs
- No `alert()`
- No `console.log` (structured logger or remove)
- Language guard (Spanish primary)

Before ship, also run:
- `npx vitest run` — the full suite (1100+ tests)
- `bash scripts/gsd-verify.sh --ratchets-only` — the ratchets

Test file convention:

```
src/lib/leads/activities.ts          → src/lib/leads/__tests__/activities.test.ts
src/app/api/leads/[id]/route.ts      → src/app/api/leads/[id]/__tests__/route.test.ts
src/components/aguila/KPITile.tsx    → src/components/aguila/__tests__/KPITile.test.tsx
```

Mandatory tests when adding:
- A `lib/` function → unit tests
- An API route → auth fence (401 for no session, 401 for wrong role),
  validation fence (400 on bad input), happy path, DB failure path
- A migration → shape test that greps the SQL
- A component with interactivity → render + event tests

## 14. The ratchet system

`scripts/gsd-verify.sh` enforces per-metric baselines that only go
down. Current ratchet list (selected):

| Ratchet | Baseline | Rule |
|---|---|---|
| Hardcoded hex in src/app | 616 | Don't add new hex |
| console.error/warn | 128 | Don't add new console calls |
| Role-from-cookie reads | 0 | Never read role from raw cookies — use `session.role` |
| /mi-cuenta red/amber strings | 0 | Calm-tone contract on client A/R surface |
| scripts/ silent .catch() | 153 | Don't add new empty-catch handlers |
| scripts/ rate hardcodes | 0 | Always route rates through `system_config` |
| Opaque glass outside primitive | 0 | Never inline glass chemistry |

When you reduce a ratchet, update its baseline in the script. When
you add a new class of violation to prevent, add a ratchet.

## 16. When NOT to ship

| You're about to… | Stop because |
|---|---|
| Push to main without tests | Every new lib/api needs tests |
| Add a feature that only works for EVCO | Multi-tenant safe from day 1 |
| Hardcode a color | Use a `--portal-*` token |
| Hardcode a client code (`'9254'`) | Use `session.companyId` |
| Add a nav item | Needs Tito + Renato IV sign-off |
| Skip RLS on a new table | Deny-by-default, always |
| Swallow a failure with empty `.catch()` | Every failure must surface |
| Render unsanitized AI output | DOMPurify on dynamic HTML |
| Amend a commit after pre-commit hook failed | Hook failure means commit didn't happen — amend would modify the PREVIOUS commit. Make a new commit instead. |
| Force-push to main | Never without explicit user request |
| Use `new Date()` on a deadline without timezone | America/Chicago required on compliance deadlines |
| Assume a migration applied on dev will apply on prod | Every migration runs via `supabase db push` against live |
