# V1.5 F8 — End-to-End Trace View · Audit

`composeTrace()` in `src/lib/trace/compose.ts` merges eight sources in parallel (`workflow_events`, `expediente_documentos`, `classification_sheets`, `pedimento_export_jobs`, `anexo_24_export_jobs`, `pece_payments`, `mve_alerts`, `quickbooks_export_jobs`) into a single `TraceEvent[]` stream sorted newest-first. Every source runs through `safeSelect()` which swallows both thrown errors and Supabase error responses, so a missing table or an RLS-denied query drops only that source and the rest of the timeline still renders. The tráfico lookup is company-scoped defense-in-depth; `pedimentoId` is resolved once up-front so PECE rows can be joined without a second round-trip per row. `/traficos/[id]/trace` is a server component gated to admin / broker / operator / contabilidad (and cliente for their own tráfico via `companyId` filter), hosts `<TraceTimeline>` with sticky per-day headers and silver dots, and emits telemetry as `page_view` with `metadata.event = 'trace_view_opened'` per the locked TelemetryEvent union. The tráfico detail header gains a "Ver cronología completa" link. **Deferred:** actor-name resolution for workflow events (only the event_type is surfaced today; mapping `payload.actor_id` → `client_users.display_name` is a follow-up), per-source filter toggles, granular cliente visibility rules for MVE alerts (currently hidden from client via RLS policies, not app logic), and per-event deep links into the originating surface. **Test delta:** +5 tests (282 → 287) covering multi-source merge + sort, missing-table null-safety, all-empty path, day bucketing, and unparseable-timestamp skip.
